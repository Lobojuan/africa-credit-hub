## 1. الملخص التنفيذي

تقدم هذه الوثيقة تقييماً شاملاً لضوابط الأمان المطبقة في نظام سجل الائتمان مقارنة بالمتطلبات المحددة في مواصفات متطلبات البرمجيات (SRS) الإصدار 1.2. يتعامل النظام مع بيانات مالية وشخصية حساسة عبر جميع الدول الأفريقية البالغ عددها 54 دولة ويدعم خمس لغات (الإنجليزية والفرنسية والبرتغالية والعربية والسواحلية) ويجب أن يمتثل لمتطلبات حماية البيانات والتنظيم المالي.

تم تنفيذ جميع متطلبات الأمان غير الوظيفية العشرة (NFR-SEC-01 إلى NFR-SEC-10)، إلى جانب خمسة عشر تحسيناً أمنياً مؤسسياً (ENT-01 إلى ENT-15) بما في ذلك المصادقة متعددة العوامل TOTP وتبادل رمز Bearer وفق OAuth 2.1 وسلاسل تجزئة سجلات التدقيق المقاومة للتلاعب والمطابقة الضبابية للكيانات وروبوت محادثة النزاعات وتحسينات النطاق الترددي المنخفض ودعم رفع XBRL وفرض الاحتفاظ بالبيانات وإدارة أسعار الصرف وإدارة واجهة برمجة التطبيقات والبحث العالمي ورفع صور/وثائق الهوية وبيئة العرض التوضيحي للمستثمرين والتحليلات المرئية للوحة المعلومات والجولة التوضيحية التفاعلية. يوضح هذا التقرير كل ضابط أمني وتنفيذه وحالة الامتثال.

## 2. ضوابط المصادقة

### 2.1 المصادقة القائمة على كلمة المرور

| الضابط | التنفيذ |
|--------|---------|
| طريقة المصادقة | اسم المستخدم/كلمة المرور مع جلسة من جانب الخادم |
| تجزئة كلمة المرور | bcryptjs مع 10 جولات ملح |
| تخزين كلمة المرور | كلمات المرور المجزأة مخزنة في عمود `users.password`؛ لا يتم تخزين أو تسجيل النص الصريح أبداً |
| نقطة نهاية تسجيل الدخول | `POST /api/auth/login` |
| نقطة نهاية تسجيل الخروج | `POST /api/auth/logout` |
| التحقق من الجلسة | `GET /api/auth/me` |

### 2.2 سياسة كلمة المرور

| السياسة | المتطلب | التنفيذ |
|---------|---------|---------|
| الحد الأدنى للطول | 8 أحرف | التحقق بالتعبير النمطي: `.{8,}` |
| حرف كبير | حرف واحد على الأقل | التعبير النمطي: `(?=.*[A-Z])` |
| حرف صغير | حرف واحد على الأقل | التعبير النمطي: `(?=.*[a-z])` |
| رقم | رقم واحد على الأقل | التعبير النمطي: `(?=.*\d)` |
| حرف خاص | حرف واحد على الأقل | التعبير النمطي: `(?=.*[!@#$%^&*()_+\-=\[\]{};':"\\|,.<>\/?])` |
| انتهاء صلاحية كلمة المرور | 90 يوماً | فحص الطابع الزمني `passwordChangedAt` عند تسجيل الدخول؛ تغيير إجباري عند انتهاء الصلاحية |
| تغيير تسجيل الدخول الأول | مطلوب عند التمييز | علم منطقي `mustChangePassword` |

**نمط التعبير النمطي الكامل:**
```regex
^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[!@#$%^&*()_+\-=\[\]{};':"\\|,.<>\/?]).{8,}$
```

### 2.3 المصادقة متعددة العوامل (MFA) — ENT-01

| الضابط | التنفيذ |
|--------|---------|
| طريقة MFA | TOTP (كلمة مرور لمرة واحدة قائمة على الوقت) عبر مكتبة `otpauth` |
| تخزين السر | عمود `mfaSecret` (varchar، قابل للقيم الفارغة) في جدول `users`؛ مخزن كسر مشفر بـ base32 |
| علم التمكين | عمود `mfaEnabled` (منطقي، افتراضي false) في جدول `users` |
| نقطة نهاية الإعداد | `POST /api/auth/mfa/setup` — ينشئ سر TOTP ويعيد URI `otpauth://` لرمز QR |
| نقطة نهاية التحقق | `POST /api/auth/mfa/verify` — يتحقق من رمز TOTP المكون من 6 أرقام ويفعّل MFA |
| نقطة نهاية التعطيل | `POST /api/auth/mfa/disable` — يعطل MFA للمستخدم المصادق عليه |
| تدفق تسجيل الدخول | `POST /api/auth/login` يعيد `{ requireMfa: true }` عندما يكون MFA مفعلاً؛ يطلب العميل رمز TOTP |
| نقطة نهاية تسجيل دخول MFA | `POST /api/auth/mfa/login` — يتحقق من رمز TOTP ويكمل المصادقة |
| مكون الواجهة | `mfa-setup.tsx` — حوار إعداد مع عرض رمز QR وإدخال التحقق |
| دعم الترجمة | ترجمات كاملة EN/FR/PT تحت مفاتيح `mfa.*` و `login.mfa*` |

**تدفق تسجيل دخول MFA:**
1. يقدم المستخدم اسم المستخدم/كلمة المرور عبر `POST /api/auth/login`
2. إذا كان `mfaEnabled = true` للمستخدم، يعيد الخادم `{ requireMfa: true, userId }` بدون إنشاء جلسة
3. يعرض العميل حقل إدخال رمز TOTP
4. يدخل المستخدم الرمز المكون من 6 أرقام من تطبيق المصادقة
5. يرسل العميل `POST /api/auth/mfa/login` مع `{ userId, code }`
6. يتحقق الخادم من رمز TOTP مقابل السر المخزن
7. إذا كان صالحاً، يتم إنشاء الجلسة ومصادقة المستخدم
8. إذا كان غير صالح، يتم إرجاع HTTP 401 مع خطأ "Invalid MFA code"

### 2.4 قفل الحساب

| الضابط | القيمة |
|--------|--------|
| الحد الأقصى للمحاولات الفاشلة | 3 |
| مدة القفل | 15 دقيقة |
| تتبع القفل | عداد `failed_login_attempts` في جدول `users` |
| انتهاء القفل | طابع زمني `locked_until` في جدول `users` |
| إعادة تعيين العداد | عند تسجيل الدخول الناجح (`resetFailedAttempts`) |
| تسجيل التدقيق | تسجيل المحاولات الفاشلة وأحداث القفل مع عنوان IP |

**تدفق القفل:**
1. يزيد تسجيل الدخول الفاشل `failed_login_attempts`
2. كل محاولة فاشلة تنشئ إدخال سجل تدقيق `LOGIN_FAILED`
3. عند 3 محاولات فاشلة، يتم تعيين `locked_until` إلى `الآن + 15 دقيقة`
4. يتم إنشاء إدخال سجل تدقيق `ACCOUNT_LOCKED`
5. محاولات تسجيل الدخول اللاحقة أثناء القفل تعيد HTTP 423
6. تسجيل الدخول الناجح يعيد تعيين العداد ويزيل القفل

### 3.2 الفرض من جانب الخادم

يتم فرض التفويض على طبقة واجهة برمجة التطبيقات باستخدام البرمجيات الوسيطة:

```typescript
function requireRole(...roles: string[]) {
  return (req, res, next) => {
    if (!req.session?.userRole || !roles.includes(req.session.userRole)) {
      return res.status(403).json({ message: "Insufficient permissions" });
    }
    next();
  };
}
```

## 4. إدارة الجلسات

### 4.1 تكوين الجلسة

| المعلمة | القيمة | ملاحظات |
|---------|--------|---------|
| مخزن الجلسات | MemoryStore | تخزين جلسات في الذاكرة مع تنظيف دوري |
| سر الجلسة | متغير البيئة `SESSION_SECRET` | يعود إلى سر التطوير في بيئة التطوير |
| Cookie HttpOnly | `true` | يمنع وصول JavaScript من جانب العميل |
| Cookie Secure | `false` (قابل للتكوين) | يجب أن يكون `true` في الإنتاج مع HTTPS |
| Cookie SameSite | `lax` | حماية CSRF |
| الحد الأقصى لعمر الجلسة | 8 ساعات | عمر الجلسة المطلق |
| مهلة الخمول | 15 دقيقة | امتثال NFR-SEC-09 |
| تنظيف المخزن | كل 24 ساعة | `checkPeriod: 86400000` |

### 4.2 تنفيذ مهلة الخمول (NFR-SEC-09)

```typescript
const IDLE_TIMEOUT_MS = 15 * 60 * 1000; // 15 دقيقة

app.use((req, res, next) => {
  if (req.session?.userId && req.session.lastActivity) {
    const idle = Date.now() - req.session.lastActivity;
    if (idle > IDLE_TIMEOUT_MS) {
      req.session.destroy(() => {});
      return res.status(440).json({ message: "Session expired due to inactivity" });
    }
  }
  if (req.session?.userId) {
    req.session.lastActivity = Date.now();
  }
  next();
});
```

- يتم تحديث الطابع الزمني `lastActivity` في كل طلب مصادق
- يتم تدمير الجلسة عندما يتجاوز وقت الخمول 15 دقيقة
- يتم إرجاع حالة HTTP 440 (مهلة تسجيل الدخول)
- تكتشف الواجهة الأمامية الحالة 440 وتعيد التوجيه إلى صفحة تسجيل الدخول

### 4.3 بيانات الجلسة

| الحقل | النوع | الوصف |
|-------|-------|-------|
| userId | string | معرف المستخدم المصادق عليه |
| userRole | string | دور المستخدم للتفويض |
| lastActivity | number | طابع زمني لآخر نشاط (بالمللي ثانية منذ العصر) |

## 5. أمان واجهة برمجة التطبيقات

### 5.1 مصادقة واجهة برمجة التطبيقات الداخلية

تتطلب جميع نقاط نهاية واجهة برمجة التطبيقات الداخلية (تحت `/api/`) مصادقة قائمة على الجلسة، باستثناء:
- `/api/auth/login`
- `/api/auth/logout`
- `/api/auth/me`
- `/api/health`
- `/api/external/*`

يشمل ذلك نقطة نهاية تصور لوحة المعلومات `GET /api/dashboard/chart-data`، التي تُعيد بيانات الاتجاه الشهري وتوزيع الحالات وتوزيع الأنواع وتوزيع الدول لوحدة التحليلات المرئية للوحة المعلومات (ENT-14). هذه النقطة محمية بالبرمجيات الوسيطة `requireAuth` وتتطلب جلسة مصادقة نشطة.

البرمجيات الوسيطة للمصادقة:
```typescript
app.use("/api", (req, res, next) => {
  if (req.path.startsWith("/auth") || req.path.startsWith("/external")) return next();
  requireAuth(req, res, next);
});
```

### 5.2 مصادقة واجهة برمجة التطبيقات الخارجية

تدعم نقاط نهاية واجهة برمجة التطبيقات الخارجية طريقتين للمصادقة:

#### 5.2.1 مصادقة X-API-Key

مصادقة مفتاح واجهة برمجة التطبيقات عبر رأس HTTP `X-API-Key` (الطريقة الأصلية):

#### 5.2.2 مصادقة رمز Bearer وفق OAuth 2.1 — ENT-04

| الضابط | التنفيذ |
|--------|---------|
| نوع المنح | `client_credentials` |
| نقطة نهاية الرمز | `POST /api/external/oauth/token` |
| تنسيق الرمز | JWT (JSON Web Token) موقع بـ HS256 |
| مكتبة الرمز | `jsonwebtoken` |
| انتهاء صلاحية الرمز | ساعة واحدة (3600 ثانية) |
| تنسيق الطلب | `{ "grant_type": "client_credentials", "client_id": "<api_key_prefix>", "client_secret": "<full_api_key>" }` |
| تنسيق الاستجابة | `{ "access_token": "<jwt>", "token_type": "Bearer", "expires_in": 3600 }` |
| الاستخدام | رأس `Authorization: Bearer <access_token>` على نقاط نهاية واجهة برمجة التطبيقات الخارجية |

**تدفق رمز OAuth 2.1:**
1. يرسل العميل `POST /api/external/oauth/token` مع بيانات اعتماد مفتاح واجهة برمجة التطبيقات
2. يتحقق الخادم من مفتاح واجهة برمجة التطبيقات عبر بحث تجزئة SHA-256
3. ينشئ الخادم JWT يحتوي على `institutionId` و `permissions` و `apiKeyId`
4. يتم توقيع JWT بسر الجلسة وإعادته إلى العميل
5. يضمن العميل JWT في رأس `Authorization: Bearer <token>` في الطلبات اللاحقة
6. يتحقق الخادم من توقيع JWT وانتهاء صلاحيته في كل طلب
7. يتم قبول كل من مصادقة X-API-Key ورمز Bearer على جميع نقاط نهاية واجهة برمجة التطبيقات الخارجية

تستخدم نقاط نهاية واجهة برمجة التطبيقات الخارجية أيضاً مصادقة مفتاح واجهة برمجة التطبيقات:

| الضابط | التنفيذ |
|--------|---------|
| رأس المصادقة | `X-API-Key` |
| تجزئة المفتاح | SHA-256 (crypto.createHash) |
| تخزين المفتاح | يتم تخزين التجزئة فقط (عمود `key_hash`)؛ يتم عرض المفتاح الكامل مرة واحدة عند الإنشاء |
| بادئة المفتاح | بادئة مرئية للتعريف (`sim_XXXXXXXX`) |
| مستويات الصلاحيات | `submit` (كتابة فقط)، `read` (قراءة فقط)، `full` (قراءة + كتابة) |
| ربط المؤسسة | كل مفتاح مرتبط بمؤسسة محددة |
| فحص حالة المؤسسة | يجب أن تكون المؤسسة `active` لكي يعمل المفتاح |
| تتبع آخر استخدام | تحديث الطابع الزمني `last_used_at` في كل استدعاء لواجهة برمجة التطبيقات |
| إلغاء المفتاح | إلغاء فوري مع طابع زمني `revoked_at` |

### 5.3 إنشاء مفتاح واجهة برمجة التطبيقات

```typescript
function generateApiKey() {
  const prefix = "sim_" + crypto.randomBytes(4).toString("hex");
  const secret = crypto.randomBytes(24).toString("hex");
  const fullKey = `${prefix}_${secret}`;
  const hash = crypto.createHash("sha256").update(fullKey).digest("hex");
  return { fullKey, prefix, hash };
}
```

- تنسيق المفتاح: `sim_XXXXXXXX_YYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYYY`
- البادئة: 8 أحرف ست عشرية للتعريف
- السر: 48 حرفاً ست عشرياً للأمان
- إجمالي الإنتروبيا: 224 بت (28 بايت عشوائي)

### 5.4 تدفق التحقق من مفتاح واجهة برمجة التطبيقات

1. استخراج رأس `X-API-Key`
2. تجزئة المفتاح بـ SHA-256
3. البحث عن التجزئة في جدول `api_keys`
4. التحقق من أن حالة المفتاح `active`
5. التحقق من أن المؤسسة المرتبطة `active`
6. فحص مستوى الصلاحيات مقابل متطلبات نقطة النهاية
7. تحديث الطابع الزمني `last_used_at`
8. المتابعة مع الطلب

### 5.5 تخفيف SSRF

تتضمن ميزة "اختبار الاتصال" في وحدة إدارة واجهة برمجة التطبيقات حماية ضد تزوير الطلبات من جانب الخادم (SSRF) لمنع إساءة استخدام طلبات HTTP الصادرة.

| الضابط | التنفيذ |
|--------|---------|
| التحقق من عنوان URL | حظر الطلبات إلى عناوين الشبكة الداخلية/الخاصة قبل إجراء الطلبات الصادرة |
| المضيفات المحظورة | `localhost`، `127.0.0.1`، `169.254.169.254` (نقطة نهاية البيانات الوصفية السحابية) |
| نطاقات IP المحظورة | نطاقات IP الخاصة: `10.x.x.x`، `192.168.x.x`، `172.16-31.x.x` |
| تقييد البروتوكول | يُسمح فقط ببروتوكولي `HTTP` و`HTTPS`؛ يتم رفض جميع المخططات الأخرى |
| التنفيذ | منطق التحقق من URL في `server/routes.ts` يتحقق من جميع عناوين URL المقدمة من المستخدم قبل إجراء الطلبات الصادرة |

**تدفق التحقق من SSRF:**
1. يقدم المستخدم عنوان URL عبر ميزة "اختبار الاتصال" في وحدة إدارة واجهة برمجة التطبيقات
2. يحلل الخادم عنوان URL ويستخرج اسم المضيف والبروتوكول
3. يتم فحص البروتوكول مقابل قائمة السماح (HTTP/HTTPS فقط)
4. يتم فحص اسم المضيف مقابل قائمة الحظر (localhost، 127.0.0.1، نقطة نهاية البيانات الوصفية)
5. يتم فحص اسم المضيف مقابل نطاقات IP الخاصة (10.x، 192.168.x، 172.16-31.x)
6. إذا فشل أي فحص، يتم رفض الطلب مع رسالة خطأ مناسبة
7. إذا نجحت جميع الفحوصات، يتم إجراء الطلب الصادر وإعادة الاستجابة إلى المستخدم

| نقل مفتاح واجهة برمجة التطبيقات | HTTPS مطلوب لواجهة برمجة التطبيقات الخارجية |

### 6.3 معالجة البيانات الحساسة

- لا يتم إرجاع كلمات المرور أبداً في استجابات واجهة برمجة التطبيقات (دالة `stripPassword`)
- يتم عرض مفاتيح واجهة برمجة التطبيقات الكاملة مرة واحدة فقط عند الإنشاء
- يتم تخزين بيانات الجلسة من جانب الخادم فقط (يحتوي الكوكي على معرف الجلسة فقط)
- يتم اقتطاع تفاصيل سجل التدقيق إلى 200 حرف في مخرجات وحدة التحكم

### 6.4 التحقق من المدخلات

يتم التحقق من جميع مدخلات المستخدم باستخدام مخططات Zod المشتقة من تعريفات جداول Drizzle ORM:

```typescript
const parsed = insertBorrowerSchema.parse(req.body);
```

- التحقق من النوع (سلسلة نصية، رقم، منطقي، قيم التعداد)
- فرض الحقول المطلوبة
- تقييد قيم التعداد
- منع حقن SQL عبر الاستعلامات المعلمة (Drizzle ORM)، بما في ذلك داخل محرك فرض الاحتفاظ ووحدات تكوين واجهة برمجة التطبيقات

| المؤسسات | APPROVE_INSTITUTION |
| مفاتيح واجهة برمجة التطبيقات | CREATE_API_KEY, REVOKE_API_KEY |

### 7.2 حقول سجل التدقيق

| الحقل | الوصف |
|-------|-------|
| id | معرف فريد لإدخال السجل |
| user_id | المستخدم الذي نفذ الإجراء |
| action | نوع الإجراء (المعدد أعلاه) |
| entity | نوع الكيان المتأثر (user, borrower, credit_account, إلخ) |
| entity_id | معرف الكيان المتأثر |

| created_at | الطابع الزمني للإجراء |

### 7.3 سلسلة التجزئة المقاومة للتلاعب — ENT-07

| الضابط | التنفيذ |
|--------|---------|
| خوارزمية التجزئة | SHA-256 (عبر وحدة `crypto` في Node.js) |
| حقول التجزئة | أعمدة `previousHash` و `currentHash` في جدول `audit_logs` |
| بناء السلسلة | `currentHash` لكل إدخال جديد = SHA-256(`previousHash` + `action` + `entity` + `details` + `timestamp`) |
| الإدخال الأول | `previousHash` = `"genesis"` للإدخال الأول في السلسلة |
| نقطة نهاية التحقق | `GET /api/audit/verify-integrity` — يتحقق من سلسلة التجزئة بالكامل |
| مؤشر الواجهة | شارة سلامة في صفحة مسار التدقيق (`data-testid="badge-integrity-status"`) |
| زر التحقق | `data-testid="button-verify-integrity"` يُشغل التحقق من السلسلة |

**عملية التحقق من سلسلة التجزئة:**
1. يستدعي العميل `GET /api/audit/verify-integrity`
2. يسترد الخادم جميع إدخالات سجل التدقيق مرتبة حسب `created_at ASC`
3. لكل إدخال، يعيد الخادم حساب `currentHash` المتوقع من `previousHash` + بيانات الإدخال
4. يتحقق الخادم من تطابق `currentHash` مع القيمة المخزنة
5. يتحقق الخادم من تطابق `previousHash` مع `currentHash` للإدخال السابق
6. يعيد `{ valid: true/false, totalEntries, checkedEntries, brokenAt }`
7. تعرض الواجهة الأمامية شارة "Valid" خضراء أو شارة "Broken" حمراء وفقاً لذلك

### 7.4 ثبات سجل التدقيق

- سجلات التدقيق للإضافة فقط (عمليات INSERT فقط)
- لا توجد نقاط نهاية UPDATE أو DELETE لسجلات التدقيق
- يتم ترتيب السجلات حسب `created_at DESC` للعرض
- يتم إرجاع آخر 200 إدخال عبر واجهة برمجة التطبيقات (مع إمكانية الترقيم)
- توفر سلسلة التجزئة دليلاً تشفيرياً على التلاعب (ENT-07)

### 7.4 تتبع عنوان IP

```typescript
app.set("trust proxy", true);
```

يتم التقاط عناوين IP في جميع العمليات المراقبة عبر `req.ip`، الذي يحترم رؤوس `X-Forwarded-For` عند وجود وكيل عكسي.

## 11. معالجة الإشارات والمرونة

ينفذ التطبيق معالجة دفاعية للإشارات:

```typescript
process.on("SIGHUP", () => { /* تجاهل إشارة الإغلاق */ });
process.on("SIGPIPE", () => { /* تجاهل الأنبوب المكسور */ });
process.on("uncaughtException", (err) => { console.error("Uncaught exception:", err); });
process.on("unhandledRejection", (err) => { console.error("Unhandled rejection:", err); });
```

- يتم تجاهل SIGHUP لمنع الإنهاء المُشغل بسير العمل
- يتم تجاهل SIGPIPE لمنع أعطال الأنبوب المكسور
- يتم تسجيل الاستثناءات غير المُلتقطة والرفض غير المُعالج لكنها لا تتسبب في تعطل العملية

## 12. القيود المعروفة والتوصيات

### 12.1 القيود الحالية

| المجال | القيد | مستوى المخاطر | التوصية |
|--------|-------|---------------|---------|
| مخزن الجلسات | MemoryStore (غير مستمر) | متوسط | استخدام Redis أو PostgreSQL لمخزن الجلسات في مجموعات الإنتاج |
| Cookie Secure | معين إلى `false` افتراضياً | عالي | تعيين `cookie.secure = true` عند تكوين HTTPS |
| تحديد المعدل | غير منفذ على نقاط نهاية واجهة برمجة التطبيقات | متوسط | إضافة برمجيات وسيطة لتحديد المعدل (مثل express-rate-limit) |
| CORS | غير مكون بشكل صريح | منخفض | تكوين رؤوس CORS إذا تم تقديم الواجهة الأمامية من أصل مختلف |
| CSP | لا توجد رؤوس سياسة أمان المحتوى | منخفض | إضافة رؤوس CSP عبر البرمجيات الوسيطة أو الوكيل العكسي |
| تاريخ كلمة المرور | غير متتبع | منخفض | تتبع تجزئات كلمات المرور السابقة لمنع إعادة الاستخدام |
| تجميع الجلسات | MemoryStore لعقدة واحدة فقط | عالي | استخدام Redis للنشر متعدد العقد |

### 12.2 توصيات تقوية الإنتاج

1. **تمكين HTTPS** وتعيين `cookie.secure = true`
2. **استبدال MemoryStore** بـ Redis لتخزين الجلسات في بيئات متعددة العقد
3. **إضافة تحديد المعدل** على نقاط نهاية تسجيل الدخول وواجهة برمجة التطبيقات
4. **تنفيذ سياسة CORS** بناءً على بنية النشر
5. **إضافة رؤوس سياسة أمان المحتوى**
6. **إعداد تجميع السجلات** (حزمة ELK، Datadog، إلخ)
7. **تمكين SSL لقاعدة البيانات** مع `sslmode=verify-full`
8. **إجراء اختبار اختراق** قبل الإطلاق
9. **تنفيذ تشفير قاعدة البيانات في حالة السكون** إذا تطلبته الولاية القضائية
10. **مراجعة وتدوير** جميع بيانات الاعتماد والأسرار الافتراضية
11. **فرض MFA** لأدوار المسؤول والمنظم في الإنتاج
12. **تدوير مفاتيح توقيع JWT الخاصة بـ OAuth** بشكل دوري

## 13. أمان دورة حياة البيانات والاحتفاظ

### 13.1 محرك فرض الاحتفاظ

يتضمن النظام محرك فرض احتفاظ تلقائي مسؤول عن تنفيذ سياسات دورة حياة البيانات وفقاً لمتطلبات تنظيمية لكل دولة.

| الضابط | التنفيذ |
|--------|---------|
| الجدولة | دورة تلقائية كل 24 ساعة؛ يعمل المحرك مرة كل 24 ساعة |
| التشغيل اليدوي | `POST /api/retention-policies/enforce` — للمسؤول فقط، محمي بـ RBAC عبر `requireRole("admin")` |
| موقع المحرك | `server/retention-enforcement.ts` |

### 13.2 سياسات الاحتفاظ لكل دولة

سياسات الاحتفاظ قابلة للتكوين لكل ولاية قضائية عبر جميع الدول الأفريقية الـ 54. تتضمن السياسات الافتراضية:

| الدولة | فترة الاحتفاظ | ملاحظات |
|--------|---------------|---------|
| غانا | 10 سنوات | متوافقة مع متطلبات قانون حماية البيانات الغاني |
| إثيوبيا | 7 سنوات | احتفاظ قياسي بالبيانات المالية |
| أوغندا | 7 سنوات | احتفاظ قياسي بالبيانات المالية |
| ليبيريا | 7 سنوات | احتفاظ قياسي بالبيانات المالية |
| دول أفريقية أخرى | قابل للتكوين | يمكن للمسؤولين تعيين سياسات لكل دولة عبر جدول retention_policies |
| سجلات التدقيق | 10 سنوات (عالمياً) | مطبقة بشكل موحد عبر جميع الولايات القضائية |

### 13.3 منع حقن SQL في محرك الاحتفاظ

يستخدم محرك فرض الاحتفاظ طبقات متعددة من الحماية ضد حقن SQL:

| الضابط | التنفيذ |
|--------|---------|
| SQL معلم | يستخدم حصرياً قوالب `sql` المُعلمة من Drizzle ORM — لا يوجد استكمال سلاسل نصية في الاستعلامات |
| التحقق من اسم الجدول | يتم التحقق من أسماء الجداول مقابل قائمة سماح مُشفرة (ثابت `VALID_TABLES`)؛ يتم رفض أي اسم جدول غير موجود في القائمة |
| التحقق من قيمة الدولة | يتم التحقق من قيم الدول مقابل مجموعة `VALID_COUNTRIES` قبل الاستخدام في الاستعلامات |
| تكامل ORM | جميع الاستعلامات مبنية عبر Drizzle ORM الذي يفرض تنفيذ الاستعلامات المعلمة |

### 13.4 تسجيل تدقيق إجراءات الاحتفاظ

جميع إجراءات فرض الاحتفاظ مسجلة بالكامل في التدقيق:

- كل عملية فرض تنشئ إدخال سجل تدقيق بنوع إجراء `RETENTION_ENFORCEMENT`
- تتضمن النتائج عدد السجلات التي تم تقييمها وحذفها والاحتفاظ بها لكل جدول ودولة
- التشغيل اليدوي يسجل المستخدم المسؤول المُبادر
- التشغيل التلقائي يسجل النظام كفاعل
- يتم تخزين تفاصيل النتائج الكاملة في حقل `details` لسجل التدقيق
